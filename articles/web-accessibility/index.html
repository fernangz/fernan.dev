Web Accessibility Basics

Create inclusive experiences that work for everyone

Introduction

Web accessibility (a11y) ensures that websites and applications are usable by everyone, including people with disabilities. Accessibility isn't just ethical—it's often legally required and benefits all users. Over 1 billion people worldwide have some form of disability.

WCAG Guidelines

The Web Content Accessibility Guidelines (WCAG) define four principles, often remembered by the acronym POUR:

  • Perceivable: Information must be presentable to users in ways they can perceive
  • Operable: Interface components must be operable by all users
  • Understandable: Information and operation must be understandable
  • Robust: Content must be robust enough to work with current and future technologies

Semantic HTML

Using proper HTML elements is the foundation of accessibility:

<!-- Bad -->
<div onclick="submit()">Submit</div>

<!-- Good -->
<button type="submit">Submit</button>

<!-- Bad -->
<div class="heading">Title</div>

<!-- Good -->
<h1>Title</h1>

Use proper heading hierarchy (h1 → h2 → h3), lists, landmarks, and form labels.

Keyboard Navigation

All interactive elements must be accessible via keyboard:

  • Ensure all links and buttons are focusable
  • Provide visible focus indicators
  • Maintain logical tab order
  • Support common keyboard shortcuts
/* Always include visible focus styles */
button:focus,
a:focus {
    outline: 0.125rem solid var(--light);
    outline-offset: 0.125rem;
}

/* Never remove outline without replacement */
button:focus {
    outline: none; /* BAD! */
}

Color and Contrast

Ensure sufficient color contrast for readability:

  • Normal text: 4.5:1 contrast ratio minimum
  • Large text (18px+ or 14px+ bold): 3:1 minimum
  • UI components: 3:1 minimum

Never use color alone to convey information—add icons, patterns, or text labels.

Images and Media

Provide alternatives for non-text content:

<!-- Informative image -->
<img src="chart.png" alt="Sales increased 20% in Q4">

<!-- Decorative image -->
<img src="decoration.png" alt="" role="presentation">

<!-- Complex image -->
<img src="infographic.png" alt="Infographic summary" aria-describedby="infographic-desc">
<p id="infographic-desc">Detailed description...</p>

Forms and Labels

Every form input needs an associated label:

<!-- Good -->
<label for="email">Email</label>
<input type="email" id="email" name="email">

<!-- Also good (implicit) -->
<label>
    Email
    <input type="email" name="email">
</label>

<!-- Provide error messages -->
<input type="email" aria-invalid="true" aria-describedby="email-error">
<span id="email-error">Please enter a valid email</span>

ARIA Attributes

Use ARIA when native HTML isn't sufficient:

  • aria-label - Provide accessible name
  • aria-labelledby - Reference visible label
  • aria-describedby - Reference description
  • aria-expanded - Indicate expanded state
  • aria-hidden - Hide from assistive tech

First rule of ARIA: Don't use ARIA if native HTML works.

Motion and Animation

Respect users' motion preferences:

@media (prefers-reduced-motion: reduce) {
    *,
    *::before,
    *::after {
        animation-duration: 0.01ms !important;
        transition-duration: 0.01ms !important;
    }
}

Avoid auto-playing videos, blinking content, and parallax effects.

Testing Accessibility

Use multiple testing methods:

  • Automated tools: axe, WAVE, Lighthouse
  • Keyboard testing: Navigate entire site with Tab/Shift+Tab
  • Screen readers: NVDA (free), VoiceOver (Mac), JAWS
  • Manual testing: Check color contrast, focus states, semantics

Conclusion

Accessibility is not a feature—it's a fundamental aspect of quality web development. By following WCAG guidelines, using semantic HTML, and testing with real users, you'll create websites that work for everyone.
