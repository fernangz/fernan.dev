Cascade of Styles

Understanding CSS cascade and specificity is crucial for writing maintainable styles

Introduction

CSS (Cascading Style Sheets) is the language used to style web pages. The "cascading" part of CSS is fundamental to how styles are applied to elements. Understanding the cascade, inheritance, and specificity will help you write more predictable and maintainable CSS.

What is the Cascade?

The cascade is the algorithm CSS uses to determine which styles to apply to an element when multiple rules could apply. It's a decision-making process that considers three main factors in this order:

  1. Importance: Whether a declaration is marked as !important or comes from user agent/user stylesheets
  2. Specificity: How specific a selector is (calculated as a weight)
  3. Source Order: The order in which rules appear (later wins when tied)

Specificity Explained

Specificity is a weight assigned to CSS selectors that determines which rule wins when multiple rules target the same element. It's calculated as a four-part value (a, b, c, d):

  • a (Inline styles): (1,0,0,0) - Styles applied directly to an element via the style attribute
  • b (ID selectors): (0,1,0,0) - Each #id adds to this column
  • c (Classes/Attributes/Pseudo-classes): (0,0,1,0) - Each .class, [attribute], or :pseudo-class
  • d (Elements/Pseudo-elements): (0,0,0,1) - Each tag name or ::pseudo-element

Specificity Examples

Here are some common selectors and their specificity values:

  • #header .nav a = (0,1,1,1) - One ID, one class, one element
  • .button.primary = (0,0,2,0) - Two classes
  • div.container p = (0,0,1,2) - One class, two elements
  • a:hover = (0,0,1,1) - One pseudo-class, one element
  • * = (0,0,0,0) - Universal selector has no specificity
  • :not(.hidden) = (0,0,1,0) - The :not() pseudo-class itself has no specificity, but its argument does

Important: Specificity is compared column by column from left to right. A single ID (0,1,0,0) will always beat any number of classes (0,0,n,0).

Source Order

When two rules have the same specificity, the one that comes later in the CSS wins. This is why the order of your CSS matters:

/* This rule comes first */
.button {
    color: blue;
}

/* This rule comes second and will win (same specificity) */
.button {
    color: red;
}

This also applies across stylesheets: styles in later-loaded stylesheets override earlier ones.

Inheritance

Some CSS properties are inherited from parent elements to their children automatically. This is useful for setting default values:

  • Inherited properties: color, font-family, font-size, font-weight, line-height, text-align, list-style, visibility
  • Non-inherited properties: border, margin, padding, background, display, position, width, height

You can control inheritance with these keywords:

  • inherit - Force a property to inherit from its parent
  • initial - Reset a property to its initial/default value
  • unset - Acts as inherit for inherited properties, initial for non-inherited
  • revert - Reverts to user agent stylesheet (browser defaults)

The !important Declaration

Adding !important to a CSS declaration gives it the highest priority, overriding normal cascade rules:

.alert {
    color: red !important;
}

However, !important should be used sparingly because:

  • It makes CSS harder to maintain and debug
  • It breaks the natural cascade, making overrides difficult
  • It can lead to specificity wars (adding more !important to override)

It's better to increase specificity or restructure your CSS when possible. Legitimate uses include utility classes and when you need to override third-party styles you can't modify.

Layered Styles (@layer)

CSS Cascade Layers (@layer) is a modern feature that gives you explicit control over cascade priority:

@layer reset, base, components, utilities;

@layer reset {
    /* Lowest priority */
    * { margin: 0; }
}

@layer utilities {
    /* Highest priority */
    .hidden { display: none; }
}

Layers provide a cleaner alternative to specificity hacks and !important.

Best Practices

  • Keep specificity low - avoid overly specific selectors like div.container ul li a
  • Use classes for styling, reserve IDs for JavaScript hooks and anchors
  • Organize your CSS logically - group related styles together
  • Consider using CSS methodologies like BEM for consistent naming
  • Avoid !important unless absolutely necessary
  • Use CSS custom properties (variables) for consistent values
  • Consider using @layer for complex projects

Conclusion

Understanding the cascade of styles is essential for writing effective CSS. By mastering specificity, inheritance, and source order, you'll be able to predict how your styles will be applied and avoid common pitfalls. Remember: the goal is to write CSS that is both effective and maintainable.
